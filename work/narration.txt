今天這支影片，我們要用 GitHub 來檢視最近一次專案變更，重點放在兩個地方：第一，是 notebook 裡的分析或實驗內容；第二，是 pipeline，也就是自動化流程本身有沒有被調整。

先看 GitHub 的變更總覽。通常我會從 commit、pull request，或 changed files 開始。這一步不是急著看每一行程式碼，而是先建立地圖：哪些檔案被動到、變更集中在哪些資料夾、這次修改比較像是內容更新、流程更新，還是兩者都有。

如果看到 notebook 被修改，要特別注意三件事。第一，資料來源有沒有變，例如讀取的檔案、路徑或查詢條件是否不同。第二，處理邏輯有沒有變，例如清理資料、篩選欄位、計算指標的方式。第三，輸出結果有沒有變，包括圖表、表格、模型輸出，或最後的解讀文字。

Notebook 的 diff 有時候不太好讀，因為它包含程式碼、輸出、metadata，甚至執行順序。這時候不要只看綠色和紅色的行數，要回到實際邏輯：這個 cell 做了什麼，輸入是什麼，輸出又會影響後面哪一步。

接著看 pipeline 的部分。Pipeline 變更通常會影響專案能不能穩定重跑。常見的改動包括觸發條件、執行環境、套件版本、資料下載、測試步驟、建置步驟，或部署與輸出位置。這些修改看起來可能只是幾行 YAML 或設定檔，但影響範圍往往比 notebook 更大。

在檢查 pipeline 時，我會先看它什麼時候會被觸發。是每次 push 都跑，還是只在 pull request、排程、或指定分支上執行。接著看環境設定，像是 Python 版本、依賴安裝方式、快取設定，以及需要的 token 或 secret。最後才看實際步驟，確認每一步的輸入和輸出是否接得起來。

如果 GitHub Actions 或其他 CI 系統有執行結果，也要一起看。綠色勾勾代表流程跑過，但不代表內容一定正確；紅色失敗則要看 log，找出是依賴問題、權限問題、資料問題，還是真的邏輯壞掉。

整理一下，這次檢視的核心方法是：先看 changed files 建立全局，再分別檢查 notebook 的分析邏輯，以及 pipeline 的自動化流程。Notebook 告訴我們內容怎麼產生，pipeline 則決定這套內容能不能穩定、重複、可靠地跑起來。

所以，下次你在 GitHub 上看到類似變更，可以照這個順序判讀：先看改了哪些檔案，再看 notebook 的資料與邏輯，接著看 pipeline 的觸發條件和執行步驟，最後用 CI 結果確認這次修改是否真的能落地。
